# Encoded principles — HumanLayer, *Writing a good CLAUDE.md*

Source: https://www.humanlayer.dev/blog/writing-a-good-claude-md (principles version: 2025-06).
These are **opinionated heuristics**, not validated constants. Surface them as overridable defaults
with their provenance; the only non-overridable rule in this skill is the human-review gate.

## Thesis
LLMs are stateless and only know what each session provides. The memory file (CLAUDE.md / AGENTS.md)
loads into **every** session, making it the highest-leverage onboarding document — so it should be
deliberately **hand-crafted**, not auto-generated. `/init` auto-generation is the named anti-pattern;
this skill is its deliberate counterweight.

## WHAT / WHY / HOW framework
- **WHAT** — tech stack + project structure (especially monorepos).
- **WHY** — the project's purpose and the role of each component (the part that *cannot* be inferred
  from code — elicit it from the user).
- **HOW** — tooling (e.g. `bun` vs `node`), verification, and the test / typecheck / build commands.

## Budgets (overridable defaults)
- Frontier LLMs follow ~**150–200 instructions** reliably. Claude Code's system prompt already uses
  ~**50**, so the memory file's usable budget is smaller — account for that when counting.
- Target the root file **< 300 lines**; HumanLayer's own is **< 60**. Shorter is better.
- Keep only **universally applicable** instructions in the root; exclude task-specific content that
  distracts when irrelevant.
- Present a violation as a **green/yellow/red dashboard** (current vs target + cost of bloat + specific
  remediation), never as a pass/fail gate.

## Anti-patterns to detect and fix
1. **Over-instruction** — stuffing every command, edge case, or style guide into the file.
2. **Style/formatting rules as prose** — "never send an LLM to do a linter's job." Route them to
   linters, formatters, and pre-commit **hooks** instead.
3. **Auto-generation** (`/init`) — hand-craft every line; do not dump scanned boilerplate.
4. **Irrelevant context** — e.g. database-schema rules loaded when working on unrelated features.
5. **Copied code snippets** — they go stale; use **`file:line` pointers** to stable sources instead.
6. **Attribution / "generated by" text** — strip it.

## Progressive disclosure — the `agent_docs/` layer
Move deep or situational material out of root into focused files under `agent_docs/`, referenced from
the root with a brief "read when needed" description. Canonical files:
- `agent_docs/building_the_project.md`
- `agent_docs/running_tests.md`
- `agent_docs/code_conventions.md`
- `agent_docs/service_architecture.md`
- `agent_docs/data_access_patterns.md` (DB / comms patterns)

**Placement rule (root vs agent_docs/):** root holds content that is **universal AND short**. Demote
anything **conditional, long, rarely-used, or reference/runbook/ADR**. **Conditionality dominates
size** — a short but niche rule still moves out. Universal-but-long → one-line summary + pointer in
root, detail in `agent_docs/`. Tiny single-file projects below real complexity need **no** `agent_docs/`.

## Alternatives the article endorses (recommend, don't inline)
- **Hooks** to run formatters/linters automatically.
- **Slash commands** that carry code guidelines separately — separating implementation from formatting
  improves results.

## Instruction taxonomy (for step 3 classification)
universal · task-specific · deterministic-tooling (→ hook/linter) · project-context · command-recipe ·
architectural-context · stale-risk. Everything not `universal` is a candidate to demote, delete, or
offload.
