rar-skills · composable agent skills

One workflow,
idea → spec → tasks → shipped PR.

A collection of ~50 Claude Code skills for design review, multi-model orchestration, and coding. They aren't a grab-bag — they compose into a single pipeline where every human decision is collected up front, and everything after the approval gate runs autonomously while still optimizing architecture, tests, simplicity, and security.

01

Compose, don't duplicate

Conductor skills are thin routers. Each behavior lives in exactly one skill and is referenced, never copied.

02

Front-load judgment

Every question an agent might ask mid-flight is answered while the human is still in the room — then encoded.

03

The contract is the boundary

A task marked ready-for-agent carries everything needed to finish it unattended.

04

Vertical slices

The unit of work cuts through every layer, is demoable alone, and ships with a machine-checkable acceptance contract.

The autonomy contract

the spine of the trio

What makes the autonomous half possible: each skill lifts what the previous one decided rather than re-deriving it. The spec names the security surface and the test seams; the planner turns them into a machine-checkable contract; the builder treats that contract as its definition of done.

The design-gate router

judgment → lookup

"Which design lens applies?" is itself a mid-flight question waiting to happen. design-gate turns it into a table: it reads the surfaces a change touches (or the slice's pre-set flags), runs at most three lenses as parallel read-only reviewers, and returns one proceed / revise verdict.

When the change touches…Lens skill(s)
Overall shape of a new system, service, or subsystemmacro-architecture · design-integrity
Module / service boundaries, cross-context integration, naming vs. business languagedomain-driven-design · macro-architecture
Business-logic structure inside one context (aggregates, invariants, events)domain-driven-design
Code-structure choices, extensibility, sprawling conditionalsdesign-patterns
Layer placement, cohesion, dependency direction, scope creeparchitecture-lens
Module / class / API interface — depth, information hiding, complexitysoftware-design-philosophy
Stored state, queues, caches, migrations, retries, concurrency, external APIsdata-systems-coding-lens
React components, hooks, contexts, rendering performancereact-performance
Architecturally significant but no single lens fits → session review + ADRarchitecture_session_review

Build engines

one task → one feature → consensus-first

The same TDD + cross-model-review core, wrapped at three scopes. Each outer engine calls the inner one — it never re-implements it.

implement-and-review
scope · one task

Splits a task into frontend (Opus subagent, Kimi reviews) and backend (Codex builds, Opus reviews) tracks in parallel worktrees, loops implement → cross-review → fix, integrates, then runs full-review. Works on a bare prompt or a Slice Contract.

applies tdd · clean-code · architecture-lens · ddd · data-systems-coding-lens · test-lens
implement-feature
scope · many tasks

Decomposes a plan with to-tasks, runs the dependency DAG (independent tasks in parallel), integrates in order, and closes with a feature-wide review focused on the cross-task seams.

calls to-tasks → implement-and-review (per task) → full-review
feature-models-roundtable
scope · consensus first

When the approach is contested: poll several models for a consensus plan, resolve open caveats with you, then hand the settled plan to implement-feature to build.

calls models-roundtable → implement-feature
